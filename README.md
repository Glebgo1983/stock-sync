# insales-mpfit-stock-sync

Интеграция inSales ↔ mpFit: заказы (создание заказа в mpFit при новом заказе inSales,
проставление статуса доставки обратно в inSales) + синхронизация остатков (актуализация
`quantity` у вариантов inSales по данным склада mpFit, чтобы товары без остатка
автоматически помечались как "нет в наличии").

Основан на существующей у заказчика интеграции заказов — логика заказов
(`api/functions.py`, `api/cdek_functions.py`, `api/pochta_functions.py`) не менялась.

## Переменные окружения

Смотрите `.env.example`. Локально создайте `.env.local` (в git не попадает).
На Vercel заведите те же переменные в Project Settings -> Environment Variables.

- `MPFIT_TOKEN` — Bearer-токен mpFit. Получить можно только через их Telegram-поддержку
  (см. `help.mpfit.ru`), self-service выдачи ключа нет.
- `INSALES_USERNAME` / `INSALES_PASSWORD` — ключ приложения inSales (Basic Auth).
- `INSALES_SHOP_DOMAIN` — домен магазина, например `myshop-dbn10.myinsales.ru`.
- `CRON_SECRET` — секрет для эндпоинта `/api/sync-stock`. Важно использовать именно
  это имя переменной: Vercel Cron Jobs сам подставляет её в заголовок
  `Authorization: Bearer $CRON_SECRET` при вызове по расписанию, без доп. настройки.

### Работа без Redis

`REDIS_URL` для **синхронизации остатков** (`/api/sync-stock`) необязателен. Если он
не задан или Redis недоступен, всё, что на него завязано, тихо отключается вместо
падения синхронизации:

- журнал (раздел "Журнал синхронизации") не пишется, `/api/sync-log` будет отдавать
  пустой результат;
- таблица ID-соответствий (раздел "Сопоставление товаров") не читается и не
  пишется — сопоставление на каждом прогоне идёт заново по артикулу (`article` в
  mpFit == `sku` в inSales), как было в первой версии интеграции;
- блокировка от параллельного запуска не действует — гонки между прогонами не
  исключены.

Сама передача остатков (mpFit → inSales) при этом работает полностью — это
единственное, что требует ТЗ буквально в п.2 и п.4; остальное — уточнения к
надёжности/аудиту (п.7, п.8) и ID-based сопоставлению (п.3), которые без Redis
просто не покрываются.

**Важно:** это не относится к существующему до меня коду заказов
(`api/functions.py`, эндпоинты `/api/create` и `/api/update`) — тот использует Redis
безусловно (кэш статусов) и без `REDIS_URL` будет падать по этим двум путям, это не
менялось.

## Синхронизация остатков

`GET /api/sync-stock` — тянет полный каталог остатков mpFit (`/v1/products/stocks`) и
список товаров inSales (`/admin/products.json`), сопоставляет их и одним
batched-запросом (`PUT /admin/products/variants_group_update.json`) обновляет
`quantity` у совпавших вариантов.

Товары inSales, чей sku не найден в ответе mpFit вообще (например, цифровые товары
или то, что не ведётся через склад mpFit), **не трогаются** — остаются с текущим
остатком. Список таких sku возвращается в ответе (`unmatched_skus`) для ручной проверки.

### Сопоставление товаров (ID-based)

Основное сопоставление идёт по внутреннему ID товара mpFit, а не по артикулу —
таблица связей хранится в Redis (`mpfit_sync:product_map`, hash: `mpfit_id ->
{insales_variant_id, insales_product_id, sku, barcode}`).

- **Первый прогон / новый товар:** связи ещё нет — товар ищется по артикулу
  (`article` в mpFit == `sku` варианта в inSales), найденная пара сохраняется в
  `mpfit_sync:product_map` для последующих прогонов.
- **Все следующие прогоны:** товар с уже сохранённой связью сопоставляется по ID,
  даже если артикул на любой из сторон успел измениться.
- Если сохранённая связь ссылается на вариант inSales, которого больше нет
  (удалили/пересоздали товар), происходит откат на поиск по артикулу и запись новой
  связи.
- Артикул сравнивается как обычная строка (без URL-кодирования), поэтому `/`, `-`,
  пробелы и другие спецсимволы обрабатываются корректно из коробки.

`mpfit_id` — предположение по имени поля (`item.get("id")` из `/products/stocks`),
не подтверждено официальной документацией mpFit (см. комментарий в
`mpfit_stock_client.py`). Если поле называется иначе или в ответе отсутствует,
матчинг для таких товаров просто не перейдёт на ID-based режим и продолжит
резолвиться по артикулу на каждом прогоне — деградация без ошибок.

В ответе `/api/sync-stock` появились поля `id_matched_variants` /
`article_matched_variants` — по ним видно, растёт ли доля ID-based совпадений от
прогона к прогону (ожидаемо расти к 100% по мере накопления связей).

### Исключённые товары

`SYNC_EXCLUDED_SKUS` / `SYNC_EXCLUDED_MPFIT_IDS` (env-переменные, список через
запятую) — товары, которые синхронизация полностью игнорирует: не читает остаток,
не пишет `quantity` в inSales, не создаёт и не использует связь в
`mpfit_sync:product_map`. Можно исключать по артикулу/sku (`SYNC_EXCLUDED_SKUS`)
или по внутреннему ID товара mpFit (`SYNC_EXCLUDED_MPFIT_IDS`, устойчиво к смене
артикула). Исключение применяется в обоих проходах сопоставления — товар с уже
сохранённой ID-based связью тоже будет пропущен, если его добавили в список позже.

Исключённые товары считаются отдельно от несопоставленных: `excluded_count` /
`excluded_skus` в ответе `/api/sync-stock`, не путать с `unmatched_count` /
`unmatched_skus` (это про товары, которых просто нет в ответе mpFit).

Изменение списка не требует правки кода — только переменной окружения в Vercel
Project Settings. Как и остальные env-переменные проекта, после изменения нужен
redeploy, чтобы значение подхватилось.

### Вебхуки остатков от mpFit (мгновенное обновление)

`POST /api/stock-webhook` — принимает событие от mpFit и, если товар уже был
сопоставлен обычной синхронизацией (есть запись в `mpfit_sync:product_map`),
сразу обновляет `quantity` этого одного варианта в inSales — без ожидания
следующего прогона по расписанию.

**Настройка на стороне mpFit:** нужно взять `MPFIT_WEBHOOK_SECRET` из своих env
и подписать на события, которые реально меняют остаток, URL вида:
```
https://<ваш-проект>.vercel.app/api/stock-webhook?secret=<MPFIT_WEBHOOK_SECRET>
```
(если UI mpFit поддерживает кастомные заголовки — можно вместо query-параметра
передавать `Authorization: Bearer <MPFIT_WEBHOOK_SECRET>`).

**Важно, что осталось непроверенным:** mpFit заявляет 30+ типов событий, но
точная структура payload для каждого и то, какие именно из них относятся к
остаткам (приёмка/отгрузка/отмена/резерв/возврат/инвентаризация/брак — весь
список из п.6 ТЗ может оказаться разными типами событий), официальной
документацией не подтверждены — их сайт `/api/` рендерится через JS и не отдал
содержимое при проверке. Перед боевым включением стоит уточнить у поддержки
mpFit:
1. какой конкретно тип(ы) события отвечает за изменение остатка;
2. содержит ли payload этого события полный объект товара со `stocks`
   (как в `products/stocks`) — от этого зависит, сработает ли обработчик.

**Как обработчик защищается от неполных/непредвиденных payload:**
- Если в событии нет массива `stocks` — событие **игнорируется**, а не
  трактуется как "остаток стал 0". Это принципиально: часть из 30+ типов
  событий (особенно про сам заказ, а не про товар) может не нести остатков
  вообще, и наивная логика могла бы случайно обнулить реальный склад.
- Если товар с этим `id` ещё не сопоставлен (нет записи в `product_map`) —
  событие тоже игнорируется; сопоставление появится на ближайшем обычном
  прогоне (через артикул), и все следующие вебхуки по этому товару после
  этого будут применяться мгновенно.
- Формат конверта payload (сам объект товара на верхнем уровне, либо
  завёрнутый в `data`/`payload`/`object`/`product`) определяется по мере
  распознавания — если у mpFit другой ключ, надо будет добавить его в
  `_extract_product_payload` (`api/mpfit_webhook.py`).

Периодический прогон (`/api/sync-stock` по расписанию) **не отключается** и
не должен — он остаётся полной сверкой-подстраховкой на случай пропущенных
или неподдержанных вебхуков, это прямое требование п.5 ТЗ. Частоту прогона
после включения вебхуков можно снизить (например, до раза в несколько часов),
но полностью убирать его не стоит.

`previous_qty` в записях журнала, созданных вебхуком, всегда `null` — чтобы не
делать лишний запрос в inSales на каждое событие ради этого поля (сравнение
до/после по-прежнему доступно из записей обычной периодической синхронизации).

### Защита от параллельного запуска

`/api/sync-stock` перед началом работы берёт распределённую блокировку в Redis
(`mpfit_sync:lock`, обычный `SET NX` с TTL через `redis.Redis.lock()`). Если
предыдущий запуск (например, от cron) ещё не завершился, новый вызов сразу
возвращает `{"skipped": true, "reason": "sync_already_running", "dry_run": ...}`
без обращения к mpFit/inSales — гонки за одновременную запись остатков исключены.

Блокировка автоматически снимается через `SYNC_LOCK_TTL_SECONDS` (по умолчанию
600 секунд), даже если процесс упал и не успел её отпустить сам — следующий
запуск не будет заблокирован навсегда.

Если Redis недоступен, блокировка не может быть применена — синхронизация в этом
случае всё равно выполняется (без защиты от гонок на этот прогон), а не падает
целиком, по той же логике деградации, что и у журнала/таблицы соответствий.

### Журнал синхронизации

Каждый боевой запуск (не `dry_run`) с непустым списком совпавших вариантов пишет
в Redis:

- `mpfit_sync:log` — список последних обновлений (до 2000 записей, новые в начале):
  дата/время, sku, ID товара mpFit (`mpfit_id`, не подтверждено официальной
  документацией — см. комментарий в `mpfit_stock_client.py`), ID варианта и товара
  inSales, остаток до/после, результат (`ok`/`error`) и текст ошибки.
- `mpfit_sync:last_run` — сводка последнего запуска (счётчики, длительность, `dry_run`).
- `mpfit_sync:last_success_at` — время завершения последнего боевого запуска.

Результат/ошибка проставляются на уровне батча inSales-запроса (до 100 вариантов за
запрос): если батч упал, все варианты в нём помечаются `error` с текстом ошибки —
inSales API не даёт более точной гранулярности за один вызов.

Если Redis недоступен или `REDIS_URL` не задан, запись журнала падает тихо (в консоль),
сама синхронизация не прерывается — остатки в inSales к этому моменту уже обновлены.

`GET /api/sync-log` — отдаёт сводку и последние записи журнала (тот же
`Authorization: Bearer $CRON_SECRET`, что и `/api/sync-stock`). Параметр `limit`
(по умолчанию 100) ограничивает число записей.

**Важно:** чтобы inSales сам скрывал товар при `quantity=0`, у товара в самом
магазине должен быть включён учёт остатка ("Ограничить продажу количеством товара
на складе" в настройках товара/магазина). Это настройка на стороне inSales, вне
зоны кода этой интеграции — стоит проверить у заказчика перед первым боевым запуском.

## Передача КИЗ (кодов маркировки) в заказы

`GET /api/sync-kiz` — отдельный процесс, не связанный с синхронизацией остатков.
Переносит коды маркировки ("Честный знак") из mpFit (`/v1/cim-codes`) в кастомное
поле заказа inSales **"КИЗ"** (`id: 134636313`, только для админки — покупателю не
видно, `show_in_result`/`show_in_checkout`/`for_buyer` выключены).

### Как это работает

1. Читает недавние заказы inSales (пагинация через `from_id`, до 500 штук за
   прогон), берёт те, у кого поле "КИЗ" ещё пусто.
2. Полностью пролистывает ленту `/v1/cim-codes` (до `SYNC_KIZ_MAX_PAGES` страниц,
   по умолчанию 50 × 200 = 10 000 кодов) — **фильтра по заказу у этого эндпоинта
   нет**, подтверждено эмпирически (параметры `filter.order_id`, `filter.order_ids`,
   `filter.ids`, простой `order_id` — все игнорируются одинаково).
3. По `mpfit order_id` из кодов резолвит `orders/list` (`filter.ids`), получает
   `number` — для заказов, созданных этой интеграцией, `number` при создании
   выставлен равным ID заказа inSales (`api/functions.py:create_order`). Заказы
   mpFit из других каналов (Wildberries и т.п.) просто не найдут совпадения — это
   ожидаемая фильтрация, а не ошибка.
4. Для найденных совпадений обрезает **последние 7 символов** каждого кода,
   склеивает несколько кодов через `", "` (если у заказа несколько единиц/позиций)
   и пишет одним PUT-запросом в `fields_values_attributes` заказа.

### Ограничения (осознанный компромисс — по запросу, без Redis)

- **Без чекпоинта.** Каждый прогон сканирует ленту `cim-codes` заново с начала —
  простой вариант, но лента только растёт (коды выдаются на все каналы mpFit, не
  только на inSales), поэтому стоимость прогона будет расти со временем.
  `SYNC_KIZ_MAX_PAGES` — не чекпоинт, а просто защита от таймаута на serverless;
  заказы, чьи коды окажутся за пределами лимита страниц, будут подхвачены на
  одном из следующих прогонов (лента не переупорядочивается, старые записи не
  теряются, только отодвигаются дальше от `last_id=0`).
- Не трогает заказы, у которых поле "КИЗ" уже заполнено — значит, если код
  оказался неверным, автоматической переустановки не будет, нужно чистить вручную.
- `created_at_min`/`created_at_max` у `/admin/orders.json` не работают (проверено
  эмпирически — узкий диапазон дат всё равно вернул полную историю заказов),
  поэтому список "заказов без КИЗ" не ограничен по дате, только по количеству
  (`max_orders=500` в `fetch_orders_missing_kiz`).

Запускать можно тем же способом, что и `/api/sync-stock` — отдельной задачей в
cron-job.org (реже, КИЗ не так срочен, например раз в 15–30 минут), с тем же
заголовком `Authorization: Bearer $CRON_SECRET`.

### Проверка перед боевым запуском

```
curl -H "Authorization: Bearer $CRON_SECRET" \
  "https://<ваш-домен-на-vercel>/api/sync-stock?dry_run=true"
```

`dry_run=true` считает и возвращает сводку (`matched_variants`, `unmatched_count`,
`unmatched_skus`, ...), но ничего не пишет в inSales. Проверьте цифры, прежде чем
включать `dry_run=false` (обычный вызов без параметра тоже пишет в inSales).

## Расписание запуска

Vercel **Hobby**-план (бесплатный) разрешает встроенным Cron Jobs запускаться не
чаще раза в сутки — расписание чаще этого при деплое даёт ошибку ("Hobby accounts
are limited to daily cron jobs") и деплой не проходит. Поэтому `vercel.json`
рассчитан на Hobby без апгрейда:

```json
{ "path": "/api/sync-stock", "schedule": "0 3 * * *" }
```

Это разовый ежедневный прогон (03:00 UTC) — он же закрывает требование ТЗ о полной
сверке минимум раз в сутки (п.5), деплоится без подписки Pro и без ошибок.

**Для частой синхронизации (раз в 15 минут и чаще)**, которая на Hobby через
встроенный Vercel Cron недоступна в принципе, — внешний бесплатный планировщик,
например [cron-job.org](https://cron-job.org): создать job на вызов
```
GET https://<ваш-проект>.vercel.app/api/sync-stock
Authorization: Bearer <CRON_SECRET>
```
с нужным интервалом (у cron-job.org на бесплатном тарифе доступен интервал от 1
минуты). Оба вызова — и встроенный ежедневный, и внешний частый — используют один
и тот же эндпоинт и секрет, друг другу не мешают (`sync_lock` из раздела выше не
даст им пересечься, если один уже выполняется).

Если позже перейдёте на Vercel Pro — можно просто заменить `"0 3 * * *"` на
`"*/15 * * * *"` в `vercel.json` и убрать внешний планировщик, встроенный cron
будет справляться сам.

## Локальный запуск

```
pip install -r requirements-dev.txt
cp .env.example .env.local   # заполнить реальными значениями
uvicorn api.index:app --reload
```

## Тесты

```
pip install -r requirements-dev.txt
pytest
```

Тесты покрывают чистую логику (подсчёт доступного остатка, включая смарт-товары, и
сопоставление sku между системами) на моковых данных — без обращения к реальным API.
